Executive Summary — GreenIntel Audit

WORK
AGAINST
THE MODEL

The GreenIntel backend works — but it hides a hard‑coded secret, debug mode enabled, zero rate limiting, and no CSRF protection. We audited every route, every fallback, every silent print(). Scroll to see what we found — and what we must fix.

Enter
01 — Secrets & Debug

SECRETS
IN
PLAIN SIGHT

GreenIntel’s app.secret_key is hard‑coded in the source, and app.run(debug=True) is left on. The interactive debugger can leak code, environment variables, and even allow remote code execution. Rule one of deployment: never ship with debug enabled. Load the secret from os.environ["SECRET_KEY"] and crash early if it’s missing.

Back Turn
02 — Logging & Errors

PRINT
IS
NOT LOGGING

Every exception in GreenIntel is silently print()ed. In production, those messages are lost. Replace them with app.logger.error(...). Structured logs let you monitor, alert, and debug without SSHing into a container. Also, never expose raw tracebacks to users — flash meaningful messages instead.

Back Turn
03 — CSRF & File Uploads

NO
CSRF
TOKENS

The /upload form accepts any CSV — but no anti‑CSRF token. An attacker could trick an admin into uploading a crafted file. Add CSRF protection (Flask‑WTF), limit file size to 5 MB via MAX_CONTENT_LENGTH, and never trust file.filename without sanitising it.

04 — Rate Limiting

OPEN
DOOR
API

GreenIntel’s /api/predict has no rate limit. Anyone can flood it with requests, consume CPU, and degrade the crop‑prediction model for all users. Add Flask‑Limiter, throttle to 10 req/min per IP, and cache identical predictions. Close the door before someone walks through it.

Back Turn
05 — Testing & Deployment

TEST
OR
GUESS

No unit tests. No CI. No production WSGI server. GreenIntel runs on app.run(), which is for development only. Move to Gunicorn, write pytest fixtures, and never guess when you can assert. The model may be smart — the pipeline must be smarter.
